# MVP v0.1 需求文档

> 本文档不是最初立项时的原始 PRD，而是基于 `master` 提交历史回溯整理出的 `MVP v0.1` 版本需求说明，用于沉淀已交付范围、统一口径、支持后续版本规划。

## 1. 版本目标

`MVP v0.1` 的目标是交付一个可被真实用户使用的多人赛事记分工具，覆盖从账号登录到房间协作记分，再到扑克轮次玩法的完整主链路。

版本目标包括：

- 支持真实用户注册、登录和会话保持
- 支持多人创建/加入房间并在同一房间内实时协作
- 支持基础记分模式与扑克轮次模式
- 支持房间状态持久化、历史记录查看与基础接口文档
- 提供可上线演示的 UI 与测试基础

## 2. 目标用户

主要用户：

- 需要现场记分的小型赛事参与者
- 以熟人局为主的牌局/轮次游戏玩家
- 希望通过网页快速建房、入房、查看分数和轮次状态的普通 C 端用户

## 3. MVP 范围

### 3.1 必须具备

- 账号注册、登录、退出
- 会话鉴权
- 大厅浏览房间
- 创建房间、加入房间
- 经典记分
- 扑克轮次流程
- 实时同步
- 历史记录
- 基础 API 文档
- 可验证的自动化测试

### 3.2 明确不在 v0.1 范围内

- 支付与结算系统
- 排行榜
- 导出报表
- 管理后台
- 多组织/多赛事空间
- 复杂权限系统
- 离线模式

## 4. 功能需求

## 4.1 账号与会话

用户应可以：

- 使用邮箱、密码、昵称注册账号
- 使用邮箱和密码登录
- 退出登录
- 刷新页面后保持登录状态

系统应：

- 使用 Cookie Session 维护会话
- 对受保护接口执行鉴权
- 在未授权时返回统一格式错误

验收标准：

- 未登录用户访问受保护能力时会被拦截
- 登录后刷新页面仍能继续使用
- 注销后会话立即失效

## 4.2 大厅与房间入口

大厅应支持：

- 查看房间列表
- 展示房间号、名称、创建者、状态、创建时间、参与人数
- 搜索房间号 / 创建者 / 房间名
- 以游客身份浏览房间列表

房间入口应支持：

- 创建密码房间
- 加入密码房间
- 已加入成员再次进入时无需重复输入密码

验收标准：

- 用户可在大厅完成建房和入房
- 已在房间中的用户可直接再次进入
- 大厅信息刷新后保持一致

## 4.3 经典记分模式

经典模式下，房间内应支持：

- 查看所有成员
- 查看每个成员当前分数
- 给其他成员加分
- 实时更新分数与记录
- 结束游戏

验收标准：

- 一个用户给分后，房间内其他用户可看到分数变化
- 记录区可看到给分历史
- 房间结束后不再允许继续进行中的操作

## 4.4 扑克轮次模式

扑克轮次房间应支持：

- 创建房间时选择扑克轮次玩法
- 房主发起第一轮与后续轮次发牌
- 每轮使用完整扑克牌堆
- 显示剩余牌数
- 当前玩家抽牌
- 玩家看牌
- 玩家亮牌 / 盖牌
- 可见手牌排序
- 未亮牌统计与应给分提示

验收标准：

- 发牌后，每位参与者获得本轮手牌
- 看牌后仅当前玩家能看到自己的可见手牌
- 双击卡牌可亮牌或盖牌
- 抽牌后剩余牌数实时变化
- 新一轮开始时才触发发牌动画

## 4.5 轮次顺序规则

扑克轮次需支持以下顺序规则：

- 按玩家号轮换
- 每轮随机
- 房主指定完整顺序
- 房主指定首位，其余顺延

系统应支持：

- 发牌前配置每位玩家发牌张数
- 在需要时配置完整顺序或首位玩家
- 预览下一轮顺序

验收标准：

- 房主能根据规则完成下一轮配置
- 顺序信息能在 UI 中被清晰展示
- 本轮顺位提示与下轮预览一致

## 4.6 实时同步

系统应支持：

- 房间状态实时刷新
- 发牌、抽牌、给分等操作对同房间用户可见
- 特定抽牌事件的动画通知

验收标准：

- 同房间内多个客户端能看到一致状态
- 发牌与抽牌具有可感知的实时反馈

## 4.7 用户中心与历史

用户中心应支持：

- 查看头像、昵称、邮箱、ID
- 查看参与历史
- 查看每个房间的最终分数、给出/收到统计
- 跳转回对应房间

验收标准：

- 已参与房间会出现在历史中
- 历史项包含房间信息、状态和统计摘要

## 4.8 API 与文档

系统应提供：

- 统一响应格式
- OpenAPI JSON
- Swagger UI 浏览入口

验收标准：

- API 文档页面可访问
- 主要认证、房间、轮次、记分接口可在文档中查看

## 4.9 UI 与交互规范

MVP v0.1 的最终交互规范应包含：

- 统一页面导航
- 左上角返回上一页
- 左上角回首页图标
- 统一标题字号
- 统一 HeroUI 组件风格
- 浅绿白主题
- 房间页卡牌高可读性

验收标准：

- 大厅、个人中心、房间页头部导航风格一致
- 原生关键下拉已迁移到 HeroUI `Select`
- 卡牌在浅色主题下仍具备清晰可读性

## 5. 非功能需求

## 5.1 技术

- 基于 Next.js 16 + App Router
- 使用 TypeScript
- 使用 Prisma + SQLite
- 使用 Socket.IO 实时同步

## 5.2 可用性

- 支持移动端和桌面端主要布局
- 表单错误应通过 UI 反馈，不使用原生 `alert`
- 主要流程不依赖复杂学习成本

## 5.3 质量

- 关键页面具备自动化测试
- 主流程在提交前可通过回归测试验证

## 6. 版本验收清单

`MVP v0.1` 完成时，应满足：

- 用户可以注册、登录并保持会话
- 可以创建和加入房间
- 可以进行经典记分
- 可以进行扑克轮次游戏
- 可以完成发牌、抽牌、看牌、亮牌/盖牌
- 可以按规则配置轮次顺序
- 房间状态可实时同步
- 用户中心和历史可用
- API 文档可访问
- 页面交互与视觉主题已统一
- 自动化测试通过

## 8. 当前版本结论

截至 `master` 当前状态，`MVP v0.1` 已达到：

- 可注册登录
- 可建房入房
- 可多人实时记分
- 可使用扑克轮次模式
- 可查看用户历史
- 可查阅 API 文档
- 有基础自动化测试保障

这意味着后续版本可以从“补齐主链路”转向“增强运营、规则扩展、稳定性和商业化能力”。
